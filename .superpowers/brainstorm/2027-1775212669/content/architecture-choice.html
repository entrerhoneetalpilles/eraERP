Architecture backend — quelle approche ?

Ce choix impacte toute la structure du projet. Il n'y a pas de mauvaise réponse, mais les trade-offs sont importants.

A

Full Next.js (App Router + Server Actions)

Un seul projet Next.js pour le back-office et le portail propriétaire. La logique métier dans des Server Actions et Route Handlers. Pas de backend séparé.

Pour

  • Démarrage ultra rapide
  • Une seule stack TypeScript
  • Déploiement simple (Vercel/Railway)

Contre

  • API non réutilisable (ex: intégrations futures)
  • Couplage fort front/back
  • Moins adapté à un gros ERP multi-équipes
B

Monorepo Turborepo — Next.js + NestJS API ⭐ Recommandé

Un monorepo avec 3 packages : apps/backoffice (Next.js), apps/portail (Next.js), apps/api (NestJS REST). Shared packages pour types et utils.

Pour

  • API REST OpenAPI 3.0 dès le départ (spec)
  • Séparation claire front/back
  • NestJS = modules, RBAC, guards natifs
  • Types partagés entre tous les apps
  • Scalable pour une équipe

Contre

  • Setup initial plus long
  • Deux serveurs à gérer en dev
C

Monorepo léger — Next.js + Express/Fastify API

Même structure que B mais avec Express ou Fastify côté API. Plus simple que NestJS, moins de conventions.

Pour

  • Plus léger que NestJS
  • Flexibilité maximale

Contre

  • Moins de structure pour un ERP complexe
  • RBAC, modules, guards à implémenter manuellement

Mon choix est B — Monorepo Turborepo + NestJS. Réponds dans le terminal pour confirmer ou choisir autrement.
